# Security Documentation

This document outlines the security approach for the k3s-lab GitOps repository.

## Secret Management

### Approach
- **No secrets stored in Git**: All sensitive data is generated randomly and stored in Kubernetes secrets
- **Random generation**: Secrets are generated using cryptographically secure random number generation
- **Kubernetes-native**: Secrets are stored using Kubernetes Secret resources
- **Automatic backup**: Optional encrypted backup of secrets for disaster recovery

### Secret Types

#### Database Passwords
- **PostgreSQL**: Random 32-byte base64 encoded password
- **Redis**: Managed by Helm chart (if enabled)

#### Application Secrets
- **Authentik Secret Key**: Random 40-byte base64 encoded key
- **JWT Tokens**: Generated automatically by applications

#### Certificates
- **SSL/TLS**: Managed by cert-manager with automatic renewal
- **Internal CA**: Generated by cert-manager

### Secret Generation

Use strong randomness for any secret material you create out-of-band:

```bash
openssl rand -base64 32  # example: application password
openssl rand -base64 40  # example: application secret key
```

### Secret Storage

Create Kubernetes Secret objects in-cluster; do not commit plaintext secrets. Example for the OpenHands app expected by `apps/openhands/deployment.yaml`:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: openhands-env
  namespace: openhands
stringData:
  # Populate with your environment-specific values, e.g. API keys
  OPENAI_API_KEY: "<your-key>"
  # Add any other variables referenced by the Deployment
```

For other applications, follow a similar pattern: create a namespaced Secret with the keys your manifests expect, and reference it via `envFrom.secretRef`.

## Security Best Practices

### 1. No Secrets in Git
- ✅ All secrets are generated and stored in Kubernetes
- ✅ No hardcoded passwords or keys in repository
- ✅ Certificate files are generated dynamically

### 2. Random Generation
- ✅ Using `openssl rand` for cryptographically secure randomness
- ✅ Different entropy levels for different secret types
- ✅ Fresh generation on each deployment

### 3. Access Control
- ✅ Kubernetes RBAC controls access to secrets
- ✅ Namespace isolation for different applications
- ✅ Service accounts with minimal required permissions

### 4. Backup and Recovery
- ✅ Optional encrypted backup with GPG
- ✅ Secrets can be regenerated if needed
- ✅ Disaster recovery procedures documented

### 5. Monitoring and Auditing
- ✅ All resources labeled for easy identification
- ✅ Flux reconciliation logs available
- ✅ Kubernetes audit logs for secret access

## Backup Strategy

Example commands to export and later restore Secrets for a namespace:

```bash
# Export all secrets from a namespace (careful with handling of the file)
kubectl get secrets -n openhands -o yaml > backup-secrets-openhands.yaml

# Restore (only into a trusted environment)
kubectl apply -f backup-secrets-openhands.yaml
```

## Disaster Recovery

### Secret Recovery
1. Rotate or regenerate affected secrets
2. Restore from backups only if rotation is not possible
3. Restart affected workloads to pick up new values

### Full Cluster Recovery
1. Restore the cluster (e.g., reinstall K3s if needed)
2. Re-bootstrap Flux to this repository if required
3. Verify deployment: ensure controllers and apps are healthy

## Monitoring

### Secret Health Checks
```bash
# Check secret status
kubectl get secrets -n openhands

# Verify secret values are set
kubectl describe secret openhands-env -n openhands
```

### Application Health
```bash
# Check application status
kubectl get pods -n authentik
kubectl logs -n authentik -l app.kubernetes.io/name=authentik

# Check Flux reconciliation
flux get all
flux logs --follow
```

## Incident Response

### Secret Compromise
1. **Immediate action**: Rotate compromised secrets
2. **Investigation**: Check audit logs for unauthorized access
3. **Recovery**: Generate new secrets and restart services
4. **Documentation**: Update incident log

### Data Breach
1. **Containment**: Isolate affected systems
2. **Assessment**: Determine scope of breach
3. **Recovery**: Restore from clean backups
4. **Lessons learned**: Update security procedures
